.NET

build VS publish

dotnet build

  • 把项目源码编译成 IL（*.dll / *.exe），产物放到 bin/<Configuration>/<TFM>/，主要用于开发的“inner-loop”。它不会做 publish 阶段的拷贝/打包/转译/修剪等发布特定操作（除非你显式配置）。

dotnet publish

  • 做编译（除非你加 –no-build），再把运行时/依赖/内容文件整理到一个可以部署的文件夹，并在发布时触发发布相关的 MSBuild 目标（Razor 编译、single-file 打包、trim、ReadyToRun、AOT 等，取决于项目属性和命令行参数）。也会生成 .deps.json、.runtimeconfig.json，这些是运行时决定如何加载依赖的重要文件。

debug vs release

  • 编译器 & MSBuild 属性
    Configuration 会影响项目文件中有条件的 PropertyGroup，常见不同点包括 Optimize（Release 常为 true）、是否定义 DEBUG 常量、以及是否生成/怎样生成 PDB（DebugType/DebugSymbols）等。Visual Studio / SDK-style 项目把这些作为不同配置的默认属性植入。

  • 调试信息（PDB）
    现代 SDK-style 项目默认使用 portable PDB，默认 DebugType 在 SDK 中是 portable（这意味着 Release 下也可能生成 PDB，取决于项目设置）。如果你想在 Release 环境仍保留 PDB（便于线上符号/崩溃分析），可以在 csproj 或命令行开启 DebugSymbols=true/DebugType=portable。

  • 运行时优化
    注意：.NET 的很多最终优化是在 JIT（运行时）阶段完成，而不是编译时。但 Release/Debug 的差别仍会影响代码可读性、局部变量保留、内联/优化提示、以及 MSBuild 在发布阶段是否启用额外功能（如 PublishTrimmed、PublishReadyToRun 等）。调试时要注意 JIT 优化会影响断点/变量显示等。

实际建议（基于上面结论）

  1. 生产环境用 Release 去 publish
    dotnet publish -c Release -r <RID> [--self-contained true/false]
    Release 是为部署/生产优化的默认路径（且 .NET9 的 dotnet publish 默认就是 Release）。

  2. 如果需要线上调试（保留 PDB）：不要用 Debug 发布到生产；用 Release 发布但保留符号，例如：
    dotnet publish -c Release -p:DebugType=portable -p:DebugSymbols=true -o ./publish
    这样既保留 Release 的行为（更贴近生产运行），又能产生 PDB 供崩溃分析/符号服务器使用。DebugType=portable 是 SDK-style 的现代默认，可用来生成跨平台 PDB。

  3. 当使用 single-file / trim / ReadyToRun / AOT 时
    知道这些通常只在 publish 阶段起作用（并且在不同配置可能有不同默认值）。先在 CI 上用 Release 做一次完整的 publish，并在预生产环境里彻底跑通（尤其对反射/动态加载依赖的代码）。如果遇到奇怪的运行时异常，检查 PublishTrimmed / PublishSingleFile 等是否启用并尝试临时关闭以定位问题。

  4. 如果在脚本里以前隐式依赖 dotnet publish 会用 Debug 输出
    注意在 net8+/net9 项目上 dotnet publish 默认已改为 Release，CI/脚本需要显式 -c Debug（如果你确实想 Debug 的 publish）或调整输出路径

  5. 调试发布问题的排查清单（遇到你说的「.NET 6 用 Debug 发布出问题」时常用的检查项）：

  • 查看 publish 目录内容（是否有 PDB、.deps.json、.runtimeconfig.json）
  • 检查是否启用了 PublishTrimmed、PublishSingleFile、PublishReadyToRun 等
    在 Debug/Release 下它们是否有不同值
  • 对于 ASP.NET Core，确认 RazorCompileOnPublish 等属性是否按预期（缺少视图通常是因为 publish 时/构建时 Razor 的编译配置不一致）
  • 尝试 dotnet publish -c Release 看问题是否消失（若消失，说明可能由 Debug/Release 的配置差异或 publish-only task 导致）。
  • 如果使用 single-file + Debug 符号出现 “image mismatch”/无法加载 PDB，尝试在发布时不要打包为 single-file 或在 Release 中保留符号以定位
