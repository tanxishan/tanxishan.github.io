在 Debian 里，一个服务可能由不同的方式运行，不是所有服务都一定挂在 systemd 下。

判断是否以 systemd 管理

systemctl 查询

  • 如果有输出并显示 Loaded、Active 等信息，就说明该服务是 systemd 管理的。
  • 如果提示 Unit not found，可能不是 systemd 管理的（或者服务名不对）。

ps 查看进程树

看 PPID（父进程 ID）是不是 systemd (PID 1)

  • 如果是，那它是 systemd 启动的。例：sshd 的 PPID=1 → systemd 管理;
  • 如果 PPID 是 init.d 脚本或 shell，那就是其它方式。

journalctl 日志

  • 能查到日志 → systemd 管理。
  • 查不到 → 可能不是。

Debian 上的服务运行方式

  1. systemd

  2. System V init 脚本 (/etc/init.d/)
    传统方式，Debian 早期一直用这个。
    启动方法：
    /etc/init.d/<服务名> start|stop|restart|status
    现在 systemd 会兼容执行这些脚本，但不是原生 systemd unit。

  3. Supervisor、PM2 等进程管理器
    用来守护某些应用（特别是 Python、Node.js、Go 写的服务）。
    常用命令：

    • supervisorctl status
    • pm2 list

  4. 直接 nohup/screen/tmux 起的后台进程
    没有统一的管理，自己用 ps -ef | grep xxx 找。
    常见：

    • nohup ./app &
    • disown

  5. cron 启动的定时任务
    在 /etc/cron.* 或 crontab -e 里。
    适合周期性执行的任务，不是真正的“服务”。

  6. docker / containerd 里运行的服务
    容器化场景下，服务不归系统 init 管理，而是容器 runtime 管理。
    查方法：

    • docker ps
    • sudo ctr containers list

最佳实践

  • 系统级服务（nginx、sshd、mysql） → 用 systemd 管理。
  • 应用级守护进程（自写脚本、Python/Node app） → 建议写成 systemd unit，别用 nohup 乱放后台，方便统一管理和日志收集。
